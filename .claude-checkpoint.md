### Root Cause #4 Details (UNRESOLVED):
The RMSNorm output (node[2]) is bit-identical when read via `.Data()` (D2H copy)
for both CPU and GPU. The sum, absmax, first5, and last5 all match. BUT the binary
dump of the GQA input (node[3]'s input = node[2]'s output) shows maxDiff=12.7 at
idx 1502. The tensor sum still matches because positive/negative diffs cancel out.

**Key evidence:**
- Individual ops pass parity: MatMul (1e-5), RMSNorm (1e-7), Softmax (1e-9), Gather (exact)
- Q4_0 GEMM and Q4_K GEMM with actual model weights pass parity (max diff ~5e-5)
- The GPU tensor's DATA (via D2H copy) is correct at the time the debug log reads it
- But when GQA's Linear layer reads the SAME tensor via GPU MatMul, it gets different values
- The divergence is row-dependent: row 0 has small diff (0.098), row 1 has massive diff (12.6)
- Disabling graph tensor pool doesn't fix it
- Disabling Q4→Q8 embedding upgrade doesn't fix it
- Disabling mmap for CPU doesn't fix it

**Hypothesis:** GPU memory aliasing — the MemPool recycles a freed GPU address that still
contains the RMSNorm output, and a subsequent operation overwrites it before GQA reads it.
OR: an async CUDA operation (not on the main stream) modifies GPU memory between RMSNorm
completion and GQA read.

**Recommended next step:** Run with `compute-sanitizer --tool memcheck` on the DGX to detect
out-of-bounds GPU memory accesses or race conditions:
```bash
compute-sanitizer --tool memcheck ./bench_tps -model ~/models/gemma3-q4km/model.gguf \
  -prompt 'Hi' -tokens 1 -device cuda -temp 0
```
This will flag any invalid GPU memory reads/writes that could explain the corruption.
