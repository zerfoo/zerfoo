# T1.5 — Forward-Pass Parameter Access Audit

Scope: every weight read executed per-batch during GPU training from
`timeseries/patchtst_gpu_train.go::trainWindowedGPU` and the functions it
invokes (`encoderForward`, plus inline patch-embed / head ops).
Paths are absolute.

## Pointer provenance (context)

- `params, _ := m.extractGPUParams()` (patchtst_gpu_train.go:582) — allocates
  fresh tensors via `clone()` (line 60-64) and `tensor.New[float32]` (line 73,
  81, 163). These are the CANONICAL weight tensors for the run.
- `paramTs := params.allParamTensors()` (patchtst_gpu_train.go:594) — a slice
  of `*tensor.TensorNumeric[float32]` pointers aliasing the exact fields of
  `params` and `params.layers[i]` (see allParamTensors at lines 178-188).
- `paramTs[i].SetData(pData)` (patchtst_gpu_train.go:1152) — routes through
  `TensorNumeric.SetData` -> `storage.Set` -> `GPUStorage.Set` ->
  `TrySet` which does a host->device memcpy into the SAME device pointer
  (`gpu_storage.go:337`). So SetData mutates in place; pointer identity is
  preserved and any later read of the same `*TensorNumeric` sees new data.
- `encoderForward(ctx, m.engine, x, params.layers, fc.layerCaches, ...)`
  (patchtst_gpu_train.go:923) passes `params.layers` BY SLICE, not by copy.
  Inside the loop `layer := &layers[li]` (patchtst_encoder.go:879) — the
  `layer.qW` etc. fields are the same `*TensorNumeric` as
  `params.layers[li].qW` and therefore the same as the corresponding
  `paramTs[i]`.
- `fc.layerWTs[li].qWT` / `fc.headWT` ARE weight-derived caches but are
  **recomputed every batch** (patchtst_gpu_train.go:855-878) from the live
  `params.headW` / `params.layers[li].{q,k,v,o,ffn1,ffn2}W`, so they track
  SetData updates with one-batch freshness equal to the rest of the pipeline.
- CUDA graph capture of the forward prefix is **disabled** —
  `canCapture = false` is hard-coded at patchtst_gpu_train.go:798, so no
  captured graph holds stale device pointers.
- `fc.layerCaches[li]` holds only activation/scratch tensors (normed1, q, k,
  v, attnProj, etc.) — no weight snapshots. Confirmed by reviewing
  `gpuBatchLayerCache` struct (patchtst_gpu_train.go:304-530) and
  `allocLayerCacheBuffers` (called from encoderForward line 882).

## Caches / snapshots considered

- `fc.layerWTs[li].{qWT,kWT,vWT,oWT,ffn1WT,ffn2WT}` and `fc.headWT`
  — weight TRANSPOSES. Recomputed every batch (train loop lines 855-878) from
  the live params. No staleness beyond the intra-batch boundary. Used ONLY in
  the backward pass (`encoderBackward`), not in `encoderForward`.
- `fc.layerCaches[li]` — contains only activation scratch, no weights.
- CUDA graph (`fwdGraph`) — disabled (canCapture=false at line 798). If it
  were enabled, captured kernels would bake device pointers; since GPUStorage
  reuses the same device pointer across SetData calls, the kernels would
  STILL see fresh data (no aliasing bug from graphs alone). Irrelevant for
  the current regression.
- `cpuParams[i]` — CPU mirror used ONLY by the AdamW update step and pushed
  back via SetData. Never read by the forward pass.

## Conclusion — counts and verdict

- Weight reads audited (unique per-layer sites counted once, scaled by
  `NLayers` where applicable): **28 distinct read sites** across the
  per-batch forward graph.
- Sites that alias `paramTs`: **28 / 28** (100%).
- Sites that do NOT alias paramTs: **0**.
- Sites reading a stale cache / snapshot of weights: **0**.

`encoderForward` does NOT populate parameters once at setup. It re-reads
`layers[li].*` (which alias `params.layers[li].*`) on every batch through
pointer indirection, so any device-side mutation performed by AdamW via
`SetData` is visible on the next forward.

### Verdict: **Leans toward E3, not E2.**

E2 (forward reads a stale cached snapshot of weights) is effectively
disproved by this audit: there is no cache of weight data between the
canonical `params.*` tensors and the engine ops. Every read traces through
a pointer that SetData has just mutated. `gpuBatchForwardCache` contains
only activation scratch and on-the-fly transposes recomputed each batch; it
does NOT hold weight snapshots.

Whatever is preventing convergence must therefore be downstream of pointer
aliasing — e.g., `GPUStorage.TrySet` silently logging a warning instead of
erroring (gpu_storage.go:338-340), the host->device copy going to the wrong
address, a stride/shape mismatch, a gradient computed against the wrong
quantity, or an AdamW math bug — i.e. a flavor of E3. Suggested follow-ups
for T1.6+: (a) after a SetData call, read back the tensor via a fresh
`.Data()` and assert it matches `cpuParams[i]`; (b) check `TrySet` return
values instead of swallowing them through `Set`; (c) verify the `paramTs[i]`
order used by AdamW matches `allParamTensors` ordering bit-for-bit with the
encoder's field access (sanity check on the 16-field layer packing at
patchtst_gpu_train.go:183-185).
