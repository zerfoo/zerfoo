# T1.6 SetData GPU semantics audit

Verdict: WRITE PATH IS SAFE. SetData on a GPU-backed tensor unconditionally
performs a host->device copy (HtoD memcpy for discrete, direct unified-memory
write for managed). No early return, no residency flag, no silent skip.

## Evidence

1. Generic SetData entry point
   - ztensor/tensor/tensor.go:268 `func (t *TensorNumeric[T]) SetData(data []T) { t.storage.Set(data) }`
   - Straight delegate to the Storage interface; no kind/device branch at the tensor level.

2. Storage interface contract
   - ztensor/tensor/storage.go:18 `Set(data []T)` (part of `Storage[T]`).
   - CPU impl: ztensor/tensor/storage.go:72 `CPUStorage.Set` just reassigns `s.data = data`.

3. GPU implementation
   - ztensor/tensor/gpu_storage.go:337 `func (s *GPUStorage[T]) Set(data []T)` → calls `TrySet`, logs on error (no panic).
   - ztensor/tensor/gpu_storage.go:278 `TrySet` branches:
     - Line 279: `SetDevice(s.deviceID)` (always).
     - Line 285: if length differs, free + re-malloc (pool or runtime).
     - Line 320-330: if `len(data) > 0`:
       - managed (unified): `copy(unsafe.Slice((*T)(devicePtr), n), data)` — direct write, no Memcpy (line 321-323).
       - discrete: `runtime.Memcpy(devicePtr, src, byteSize, MemcpyHostToDevice)` — HtoD (line 325-327). This is the CUDA path; runtime is `gpuapi.Runtime`, which on CUDA backend wraps `cuda.MemcpyHtoD`.
   - There is NO early return based on any residency flag, dirty bit, shape match, etc. The only "skip" is `len(data) == 0`, which is correct.

4. Trace for "paramTs[i] is GPU tensor, SetData(cpuFloat32Slice)":
   tensor.go:268 SetData -> gpu_storage.go:337 Set -> :278 TrySet ->
   :279 SetDevice -> (length equal path) -> :320 len(data)>0 ->
   :325 Memcpy(HostToDevice) with unsafe.SliceData(data) as src and devicePtr as dst.
   Byte count = len(data)*sizeof(T). On CUDA backend this is cuMemcpyHtoD (via gpuapi adapter).

5. Data() on GPU tensor
   - ztensor/tensor/tensor.go:216 `Data()` → non-view path returns `t.storage.Slice()`.
   - ztensor/tensor/gpu_storage.go:241 `Slice()` → `TrySlice()` at :215.
   - TrySlice (managed, :222-227): allocates fresh host slice, `copy` from unified pointer.
   - TrySlice (discrete, :229-236): allocates fresh host slice, `runtime.Memcpy(... MemcpyDeviceToHost)`.
   - No caching: every Data() call allocates a brand-new host slice and re-copies from device. There is NO stale-cache risk.

## Residency / flag audit

- `GPUStorage` struct (gpu_storage.go:25) has no "dirty" or "host_valid" flag.
- `managed` only selects between unified direct-write and Memcpy; both still write on every Set.
- `view` / `refcount` affect Free(), not Set().
- No early return in TrySet other than the (implicit) len==0 guard.

## Immutable storages (orthogonal)

Several quantized storages panic on Set (Q4, Q8, Q4K, Q5K, Q6K, AWQ, GPTQ,
W8A8, IQ3S, IQ4NL, Q5_0) — see gpu-related paths in the grep at gpu_storage
audit step. These are CPU-side packed weights, not GPU residency. Not
relevant to the training write-back path (optimizers write float32 params).
mmap_storage.Set is a no-op (:136) — applies only to memory-mapped weight
files, not training state.

## Conclusion for E2 vs E3

E2 (SetData silently skipping on GPU) is REFUTED. The GPU write path is
unconditional and correct. If optimizer updates are not landing on device,
the bug is upstream of SetData: either (a) the optimizer writes to a
DIFFERENT tensor object than the forward pass reads (aliasing / parameter
identity mismatch) or (b) a stale host snapshot is being passed back that
clobbers a newer on-device state (order-of-operations between optimizer
and D2H/H2D).

Recommendation: lean toward **E3** (parameter aliasing / wrong tensor being
updated, or host snapshot round-trip clobber) rather than E2 (SetData skip).
Next step is T1.x to verify paramTs identity between forward graph and
optimizer update list, and to log devicePtr before/after SetData on a
canary step.
