# T1.4 — extractGPUParams aliasing audit

**Verdict:** E2 REFUTED. Evidence supports E3 (write-path fix).

## Key findings

1. `extractGPUParams` (`timeseries/patchtst_gpu_train.go:51-169`) clones
   every model parameter into a fresh tensor via `clone` closure
   (`:60-64`, host `make`+`copy`+`tensor.New`) and `reshapeBias`
   (`:171-175`). Therefore `gpuParams.*` tensors are SEPARATE COPIES of
   `m.patchEmb.weights`, `m.layers[i].qProj.weights`, `m.head.weights`,
   etc. No pointer aliasing between `m.*` and `params.*`.

2. `allParamTensors()` (`:178-188`) returns LIVE pointers to the
   `gpuParams` fields. So `paramTs[i] == params.headW` (and the layer
   equivalents) — same `*TensorNumeric` instance, not a copy.

3. Forward/backward inside `trainWindowedGPU` reads from `params.*`:
   - `engine.Transpose(ctx, params.headW, …, fc.headWT)` at `:855`
   - `layer := &params.layers[li]` at `:859`
   - `engine.MatMul(ctx, fc.flatInput, params.headW, fc.headOut)` at `:940`
   - Nothing in the loop touches `m.head.weights` / `m.layers[i].qProj.*`.
     `m.*` is only written by `writeBackF32FromGPU` at `:1170, 1178-1201`
     AFTER the loop.
   → Read-path aliasing is consistent: the forward pass reads from the
     SAME tensor instances that AdamW writes to.

4. CPU-mirror setup at `:609-623` is explicit: for GPU tensors, `pt.Data()`
   returns a "fresh device→host memcpy each call, not a pointer to live
   memory". `cpuParams[i]` is snapshotted once at training start.

5. AdamW step at `:1113-1153` runs on the CPU mirror and pushes back via
   `paramTs[i].SetData(pData)` at `:1152`. `SetData` → `storage.Set`
   (`ztensor/tensor/tensor.go:268-270`).

## Why E2 is refuted

E2 ("forward reads stale weights because paramTs is a copy of model
parameters, not the live tensor") would require a second layer of
copying between `paramTs[i]` and what the forward pass consumes. None
exists — both are the same `*TensorNumeric` instance (the `params.*`
field itself). The "copy" in `extractGPUParams` is relative to
`m.patchEmb.*`, not relative to the forward pass's view.

## Why E3 is the likely root cause

The write path goes: CPU AdamW → `paramTs[i].SetData(pData)` →
`storage.Set` → (hopefully) host→device memcpy. Three failure modes to
chase, in priority order:

(a) **`storage.Set` re-allocates device memory** instead of writing into
    the original device pointer. The CUDA graph captured at `:855-940`
    (guarded by `fwdCaptured`, released at `:1166`) would then keep
    pointing at the original buffer, and parameter updates would never
    reach the executing kernels.

(b) **Pre-transposed buffers baked into the captured graph.** The forward
    uses `fc.headWT` and `layerWTs` as pre-computed weight transposes
    (`:540-542`, materialized around `:855-859`). If transpose is
    computed once before capture and not re-run inside the captured
    region, updating `params.headW` has no effect until the transpose
    is re-executed.

(c) **GPU storage uses a host staging buffer** that `SetData` writes to
    without triggering a host→device flush. Consistent with the comment
    at `:609-615` that `Data()` returns a fresh memcpy (suggesting a
    host staging path exists).

## Recommendation

Pursue E3. Fast diagnostic: force `fwdCaptured = false` and re-run the
28K×20×10 bench. If loss starts moving, the bug is in the captured-graph
+ `SetData` interaction. Then narrow to (a)/(b)/(c) by inspecting the
GPU `storage.Set` implementation and the transpose-buffer lifecycle
around the CUDA graph capture.
